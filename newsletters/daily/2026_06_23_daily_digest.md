<h2 style="margin: 0 0 8px; font-size: 24px; font-weight: 800; color: #f5f5f0; line-height: 1.2;">Oak — version control built for AI coding agents</h2>

<p style="margin: 0 0 20px; font-size: 15px; color: #8a8a85; line-height: 1.5;">A new version control system that treats agents as the primary user — lazy mounts, branch-per-task, and Git interop.</p>

<p style="margin: 0 0 16px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
<strong style="color: #f7ff00;">What is it?</strong><br>
Oak is a version control system designed for autonomous coding agents. Instead of full clones and per-commit messages, Oak hydrates a repo on demand, gives every agent task its own branch, and uses the branch description as the commit message. The CLI is free and open source, with a paid hosted plan on <a href="https://oak.space" style="color: #f7ff00;">oak.space</a>.
</p>

<p style="margin: 0 0 16px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
<strong style="color: #f7ff00;">How does it work?</strong><br>
Built in Rust around BLAKE3 hashing and content-defined chunking, Oak mounts a repo by streaming the manifest first, then fetching file contents on first read — so agents start editing within a second. Benchmarks show 90–95% faster snapshots and diffs versus Git. An export command replays branches into a standard Git history for interop.
</p>

<p style="margin: 0 0 16px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
<strong style="color: #f7ff00;">Why does it matter?</strong><br>
Most VCS friction in agentic workflows comes from full clones, locked working trees, and per-commit message overhead. Oak removes those by design. The Show HN launch on June 22 hit the front page with 151 points within hours, signalling real demand from teams running coding-agent fleets.
</p>

<p style="margin: 0 0 20px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
<strong style="color: #f7ff00;">Who is it for?</strong><br>
Teams running fleets of coding agents — Codex, Claude Code, Cursor — on large or binary-heavy repos. Try it with <code style="font-family: Menlo, Consolas, monospace;">curl -fsSL https://oak.space | sh</code>.
</p>

<h2 style="margin: 0 0 8px; font-size: 24px; font-weight: 800; color: #f5f5f0; line-height: 1.2;">OpenAI Codex — SSD-burning SQLite log bug patched after 640 TB/year reports</h2>

<p style="margin: 0 0 20px; font-size: 15px; color: #8a8a85; line-height: 1.5;">OpenAI cut about 85% of Codex CLI's local log writes after a SQLite sink at TRACE level was found burning through users' SSDs.</p>

<p style="margin: 0 0 16px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
<strong style="color: #f7ff00;">What is it?</strong><br>
OpenAI Codex is the company's open-source terminal coding agent. Issue <a href="https://github.com/openai/codex/issues/28224" style="color: #f7ff00;">#28224</a> reported its local SQLite feedback log was writing roughly 37 TB in 21 days of normal use — on track for 640 TB per year. Two patches merged on June 22 cut about 85% of those writes.
</p>

<p style="margin: 0 0 16px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
<strong style="color: #f7ff00;">How does it work?</strong><br>
The log sink was installed at a default trace level, so it persisted every dependency target plus raw WebSocket and SSE payloads from each model call. PR <a href="https://github.com/openai/codex/pull/29432" style="color: #f7ff00;">#29432</a> stops logging every Responses WebSocket event; PR <a href="https://github.com/openai/codex/pull/29457" style="color: #f7ff00;">#29457</a> filters noisy targets from the persistent store.
</p>

<p style="margin: 0 0 16px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
<strong style="color: #f7ff00;">Why does it matter?</strong><br>
Consumer SSDs are rated for a few hundred terabytes of writes over their lifetime. At the measured rate, a Codex user on a 1 TB drive could burn through a full drive's endurance in months. The fix keeps local agent history without trading the disk for it.
</p>

<p style="margin: 0 0 20px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
<strong style="color: #f7ff00;">Who is it for?</strong><br>
Codex CLI users — especially heavy daily users on laptop SSDs. Pull the latest main branch and rebuild to pick up both fixes.
</p>

<h2 style="margin: 0 0 8px; font-size: 24px; font-weight: 800; color: #f5f5f0; line-height: 1.2;">Baidu Unlimited-OCR — 3B vision model parses long documents in one pass</h2>

<p style="margin: 0 0 20px; font-size: 15px; color: #8a8a85; line-height: 1.5;">Baidu's open 3B OCR model swaps standard attention for R-SWA so it can transcribe dozens of pages without the usual KV-cache blowup.</p>

<p style="margin: 0 0 16px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
<strong style="color: #f7ff00;">What is it?</strong><br>
Unlimited-OCR is a 3B open-weight vision-language model from Baidu, released under MIT with code and weights on <a href="https://huggingface.co/baidu/Unlimited-OCR" style="color: #f7ff00;">Hugging Face</a>. It parses single images, multi-page documents, and PDFs as one job, with outputs up to 32,768 tokens.
</p>

<p style="margin: 0 0 16px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
<strong style="color: #f7ff00;">How does it work?</strong><br>
Reference Sliding Window Attention (R-SWA) replaces the decoder's standard attention so the KV cache stays a constant size as the output grows. Two modes ship: a 'gundam' setting for dense single images and a 'base' mode for multi-page documents — both compatible with Hugging Face Transformers or SGLang.
</p>

<p style="margin: 0 0 16px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
<strong style="color: #f7ff00;">Why does it matter?</strong><br>
Most OCR stacks force callers to slice a document into pages, run inference per page, then stitch results back together. Unlimited-OCR turns long-document transcription into a single forward pass, simplifying pipelines for archives, contracts, invoices, and any workload where chunking loses cross-page context.
</p>

<p style="margin: 0 0 20px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
<strong style="color: #f7ff00;">Who is it for?</strong><br>
OCR engineers, document-AI teams, and researchers building long-output decoders. Install with <code style="font-family: Menlo, Consolas, monospace;">pip install transformers</code> and clone from <a href="https://github.com/baidu/Unlimited-OCR" style="color: #f7ff00;">github.com/baidu/Unlimited-OCR</a>.
</p>

</p>

<p style="margin: 0 0 16px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">

</p>

<p style="margin: 0 0 16px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
<strong style="color: #f7ff00;">Why does it matter?</strong><br>
Production OCR pipelines must fit devices ranging from phones to GPU servers, and most teams pick one model and live with the trade-off. PP-OCRv6 ships three drop-in tiers so the same toolkit covers edge, mid-tier, and server inference without retraining or stitching different vendors together.
</p>

<p style="margin: 0 0 20px; font-size: 15px; color: #f5f5f0; line-height: 1.6;">
